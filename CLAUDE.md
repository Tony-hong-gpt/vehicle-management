# 교회 통합 차량 관리 시스템 (vehicle-system)

## 프로젝트 개요

교회 보유 차량(대형버스·중형버스·12인승·7인승·승용차)을 전담 관리자 1~2명이
스마트폰으로 주간 단위로 기록하고, 상위 관리자(총무처·목회지원실)가
PC 웹 대시보드에서 실시간으로 관제하는 시스템.

**핵심 원칙**
- 일일 점검 없음 — 주 1회 주간 점검 + 이벤트 발생 시 즉시 등록
- 현장 관리자도 알림 수신, 캘린더·통계 열람, 알림 기준 커스터마이징 가능
- 상위 관리자는 현장 관리자 권한 전체 + 시스템 설정·전체 관제 추가

## 사용자 권한 체계

### 현장 관리자 (스마트폰 모바일 웹)

| 구분 | 상세 |
|------|------|
| ✅ 입력 | 주간 점검 체크리스트 / 소모품 교환 즉시 등록 / 파손·이벤트 즉시 등록 / 법정 갱신 증빙 업로드 |
| ✅ 열람 | 4색 신호등 카드뷰 / 정비 이력 타임라인 / 법정 만기 캘린더 (읽기 전용) / 통계 차트 (읽기 전용) |
| ✅ 알림 수신 | 소모품 교환 임박 (앱 푸시·인앱) / 주간 점검 리마인더 / 법정 만기 임박 |
| ✅ 알림 기준 조정 | 담당 차량의 소모품별 교환 km·기간 기준 수정 (하한선 범위 내) |
| ❌ 불가 | 시스템 설정 / 전체 차량 일괄 기준 변경 / 타 관리자 차량 설정 수정 |

### 상위 관리자 (PC 웹 + 스마트폰)

현장 관리자 모든 권한 포함 +
- 전체 차량 신호등 관제 / 주간 점검 전체 제출 현황판
- 소모품 알림 기준 기본값 일괄 설정 및 초기화
- 알림 기준 변경 이력 감사 조회
- 상위 관리자 전용 SMS 알림 수신 (🟣 법정 위반 시)

## 알림 시스템 — 소모품 단계별 발동 기준

> 모든 기준은 consumable_thresholds에서 커스텀 값 우선 조회, 없으면 consumable_defaults 사용

| 단계 | 발동 조건 | 발송 내용 |
|------|-----------|-----------|
| 1단계 사전 예고 | 커스텀 기준의 80% 도달 | "교환까지 X,000km 남았습니다" |
| 2단계 주의 🟡 | 커스텀 기준의 90% 또는 1개월 이내 | "교환 임박 — X주 또는 Xkm 이내" |
| 3단계 경고 🔴 | 기준 초과 | "즉시 교환 필요!" |

**법정 항목 알림 기준**

| 항목 | 1단계 (사전 예고) | 2단계 🟡 | 3단계 🟣 |
|------|-----------------|---------|---------|
| 보험 만기 | 60일 전 | 30일 전 | 15일 이내·만료 |
| 정기검사 만기 | 60일 전 | 30일 전 | 도달·경과 |
| 자동차세 | 30일 전 | 14일 전 | 기한 경과 |
| 운전자 면허 | 60일 전 | 30일 전 | 만기 경과 |

## 알림 기준 커스터마이징

```
현장 관리자 조정 가능 범위:
- 소모품별 교환 km 기준 (예: 엔진오일 기본 10,000km → 8,000km로 조정)
- 소모품별 교환 기간 기준 (예: 6개월 → 5개월로 조정)
- 하한선 제한: 권장값의 50% 미만 설정 불가 (서버 사이드 검증)
- 변경 시 threshold_change_log에 자동 감사 기록

상위 관리자 추가 권한:
- 전체 차량 기본값(consumable_defaults) 일괄 수정
- 특정 차량 커스텀 기준 초기화 (기본값으로 복구)
```

## 데이터베이스 테이블 (10개)

```sql
vehicles               -- 차량 마스터 (법정 필드 포함)
weekly_inspections     -- 주간 점검 기록 (일일 점검 없음)
maintenance_records    -- 소모품 교환 즉시 등록 이력
incident_records       -- 파손·방전 즉시 등록 이벤트
legal_documents        -- 보험·검사·세금 증빙 문서
consumable_defaults    -- 소모품 기본 알림 기준 및 하한선
consumable_thresholds  -- 차량별 커스텀 알림 기준
threshold_change_log   -- 알림 기준 변경 감사 로그
drivers                -- 운전자 면허 관리
notifications          -- 알림 피드 저장 (user_id / type / is_read)
```

**핵심 쿼리 패턴**

```sql
-- 소모품 잔여 주기 계산 (커스텀 기준 우선)
SELECT
  COALESCE(ct.custom_km, cd.default_km) AS threshold_km,
  COALESCE(ct.custom_months, cd.default_months) AS threshold_months,
  mr.mileage_at,
  mr.service_date,
  (v.current_mileage - mr.mileage_at) AS used_km,
  EXTRACT(MONTH FROM AGE(NOW(), mr.service_date)) AS used_months
FROM maintenance_records mr
JOIN consumable_defaults cd ON cd.part_id = mr.part_id
LEFT JOIN consumable_thresholds ct ON ct.vehicle_id = mr.vehicle_id AND ct.part_id = mr.part_id
JOIN vehicles v ON v.id = mr.vehicle_id
WHERE mr.vehicle_id = $1
ORDER BY mr.service_date DESC;

-- 현장 관리자 통계 차트 (담당 차량 범위만)
SELECT * FROM weekly_inspections
WHERE vehicle_id = ANY($assigned_vehicle_ids)
ORDER BY week_start DESC LIMIT 12;
```

  외관 점검:       주 1회 (주간 점검 항목)

1차 개발(SLICE 01~10)은 기능 구현 중심으로 진행하며, 전체 기능이 완성된 후 아래 항목을 기준으로 UI/UX를 수정·보완한다.

| 영역 | 개선 방향 |
|------|---------|
| 디자인 일관성 | 색상·타이포·간격·컴포넌트 스타일 전체 통일 |
| 모바일 최적화 | 터치 영역·폰트 크기·스크롤 동작 현장 기기 기준 재검토 |
| 빈 상태 화면 | 데이터 없을 때 안내 메시지·일러스트 추가 |
| 로딩 상태 | 스켈레톤 UI 또는 로딩 스피너 적용 |
| 에러 상태 | 네트워크 오류·권한 오류 등 사용자 친화적 메시지 |
| 네비게이션 | 탭 구조·진입 경로 사용성 테스트 후 재조정 |
| 접근성 | 버튼 레이블·명암비·키보드 접근성 기본 수준 확보 |
| 애니메이션 | 페이지 전환·피드백 인터랙션 자연스럽게 보완 |

## 개발 환경

- **로컬 Supabase**: Docker Desktop + WSL
  - API: 포트 54341 / Studio: 54343 / Mailpit: 54344
- **GitHub**: `https://github.com/Tony-hong-gpt/vehicle-system`
- **배포**: Vercel (main 브랜치 push 시 자동 배포)
- **패키지 매니저**: npm

## 새 PC에서 개발 환경 셋업

### 1. 사전 설치 (없는 경우만)

```bash
# Node.js 22 LTS 설치 확인
node -v   # v22.x.x 이상 필요

# Git 설치 확인
git --version

# Docker Desktop 설치 필요 (Supabase 로컬 실행용)
# https://www.docker.com/products/docker-desktop/

# Supabase CLI 설치
npm install -g supabase
```

### 2. 프로젝트 Clone

```bash
git clone https://github.com/Tony-hong-gpt/vehicle-system.git
cd vehicle-system
npm install
```

### 3. 환경변수 설정

프로젝트 루트에 `.env.local` 파일 생성:

```bash
# 로컬 Supabase 주소 (supabase start 실행 후 터미널에 표시되는 값으로 교체)
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54341
NEXT_PUBLIC_SUPABASE_ANON_KEY=<anon key 값>
```

### 4. Supabase 로컬 실행

```bash
# Docker Desktop이 실행 중인 상태에서
supabase start

# 실행 완료 후 출력되는 값을 .env.local에 입력:
# API URL     → NEXT_PUBLIC_SUPABASE_URL
# anon key    → NEXT_PUBLIC_SUPABASE_ANON_KEY
```

Supabase Studio: http://localhost:54343

### 5. DB 초기화 (처음 셋업 시 1회)

```bash
# 마이그레이션 + 시드 데이터 적용
supabase db reset
```

Supabase Studio → SQL Editor에서 추가 마이그레이션 수동 실행:
- `supabase/migrations/004_add_missing_consumables.sql`

### 6. 개발 서버 실행

```bash
npm run dev
# → http://localhost:3000
```

### 7. Claude Code 프로젝트 폴더 지정

새 PC에서 Claude Code 실행 시 PC마다 경로가 다르므로,
Claude Code를 열고 **프로젝트 폴더를 clone한 경로로 지정**한 후 시작.
